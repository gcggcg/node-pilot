# 多服务器任务单线程和多线程切换

## 需求概述

在任务管理页面中创建和编辑任务时，当选择多服务器执行时，添加执行模式选项：**并发执行** 或 **单线程执行**。

## 详细需求

### 1. 执行模式说明

| 模式        | 行为描述                                           | 默认值 |
|-----------|------------------------------------------------|-----|
| **并发执行**  | 每个服务器使用独立的 goroutine 协程同时执行脚本任务                | ✅ 是 |
| **单线程执行** | 按服务器添加顺序依次执行，一个服务器执行完成后才开始下一个；任意服务器执行失败则终止后续执行 | 否   |

### 2. 单线程执行逻辑

```
服务器列表: [S1, S2, S3, S4]  (按添加顺序)

执行流程:
1. 在 S1 上执行所有脚本
   - 若 S1 成功 → 继续在 S2 上执行
   - 若 S1 失败 → 终止任务，S3、S4 不再执行
2. 在 S2 上执行所有脚本
   - 若 S2 成功 → 继续在 S3 上执行
   - 若 S2 失败 → 终止任务，S4 不再执行
3. 在 S3 上执行所有脚本
   - 若 S3 成功 → 继续在 S4 上执行
   - 若 S3 失败 → 终止任务
4. 在 S4 上执行所有脚本
   - 若 S4 成功 → 任务完成
   - 若 S4 失败 → 任务失败
```

### 3. WebSocket 回显保持不变

- 每个服务器保持独立的输出窗口
- `server_start`、`output`、`server_done` 消息按服务器发送
- 前端无需修改 WebSocket 消息处理逻辑

### 3.3 TaskExecutor Service 变更

在 `backend/internal/service/task.go` 中修改 `ExecuteScript` 方法：

```go
func (e *TaskExecutor) ExecuteScript(task *model.Task, script *model.Script, servers []*model.Server, password string, serverTaskServerMap map[int64]int64) {
// ... 前置逻辑不变 ...

executionMode := task.ExecutionMode
if executionMode == "" {
executionMode = "concurrent" // 默认并发
}

if executionMode == "sequential" {
e.executeSequential(task, scripts, servers, password, serverTaskServerMap)
} else {
// 原有的并发执行逻辑
e.executeConcurrent(task, scripts, servers, password, serverTaskServerMap)
}
}

// 新增：单线程顺序执行
func (e *TaskExecutor) executeSequential(task *model.Task, scripts []*model.Script, servers []*model.Server, password string, serverTaskServerMap map[int64]int64) {
for i, srv := range servers {
if e.IsTaskCancelled(task.ID) {
// 任务被取消，停止执行
break
}

// 在单个服务器上执行所有脚本
success := e.executeScriptsOnServer(task, srv, scripts, password, serverTaskServerMap)

if !success {
// 单线程模式：服务器执行失败，终止后续服务器
logger.Debug("[TASK-%d] 单线程模式：服务器 %s 执行失败，终止后续服务器", task.ID, srv.Name)

// 标记剩余服务器为 skipped 或 cancelled
for j := i + 1; j < len(servers); j++ {
remainingSrv := servers[j]
if tsID, ok := serverTaskServerMap[remainingSrv.ID]; ok {
finished := time.Now()
e.repo.UpdateTaskServerStatus(tsID, "skipped", "", "前置服务器执行失败，跳过", nil, &finished)
}
}
break
}
}

// 更新任务状态
e.finalizeTask(task, servers)
}

// 原有的 executeBatch 改名为 executeConcurrent
func (e *TaskExecutor) executeConcurrent(task *model.Task, scripts []*model.Script, servers []*model.Server, password string, offset, total int, serverTaskServerMap map[int64]int64) {

}
```

## 验收标准

### 功能验收

- [ ] 创建任务时可选择执行模式（并发/单线程）
- [ ] 编辑任务时可修改执行模式
- [ ] 单线程模式下，服务器按添加顺序依次执行
- [ ] 单线程模式下，任意服务器执行失败时，后续服务器不再执行
- [ ] 并发模式下行为与修改前一致（默认行为）
- [ ] WebSocket 回显正常，每个服务器有独立的输出窗口
- [ ] 任务列表正确显示 execution_mode

### 边界情况

- [ ] 单服务器任务无论选择哪种模式都正常执行
- [ ] 空服务器列表任务创建成功但执行时提示错误
- [ ] 任务执行中途切换执行模式不影响正在执行的任务（基于任务创建时的模式）

### 兼容性

- [ ] 历史任务（无 execution_mode 字段）默认使用并发模式
- [ ] 数据库迁移不影响现有数据

## 注意事项

1. **并发控制**：单线程模式不需要 `sync.WaitGroup`，使用普通 for 循环即可
2. **失败处理**：单线程模式需要在 `executeScriptsOnServer` 返回 false 时立即终止
3. **取消处理**：即使在单线程模式，也要检查 `IsTaskCancelled()` 允许用户取消
4. **日志**：单线程模式添加明确日志说明当前执行到第几个服务器
5. **状态更新**：失败后剩余服务器标记为 `skipped` 状态而非 `cancelled`
